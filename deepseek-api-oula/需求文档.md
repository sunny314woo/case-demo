# DeepSeek FastAPI 教学 Demo 需求文档

## 1. 文档说明

本文档用于说明“DeepSeek FastAPI 教学 Demo”项目的建设目标、功能范围、技术约束、运行环境和验收标准，作为项目实现与课堂演示的依据。

## 2. 项目背景

本项目是一个面向中等职业学校课堂教学的 AI Web Demo，主要用于演示如何在网页中调用大语言模型接口，实现文本改写与风格生成。

项目强调以下原则：

1. 代码简单，适合课堂讲解
2. 功能聚焦，不做过度设计
3. 部署方便，适合单机演示
4. Prompt 可见、可改，突出教学价值

## 3. 项目目标

### 3.1 总体目标

构建一个可直接运行的 Web 教学示例，让教师能够在课堂中稳定演示以下内容：

1. 用户输入文本后如何提交到后端
2. 后端如何拼接 Prompt
3. 后端如何调用 DeepSeek API
4. Prompt 变化如何影响模型输出结果

### 3.2 教学目标

本项目应支持教师围绕以下知识点开展教学：

1. 认识 Web 应用的基本结构
2. 理解前端表单与后端接口的关系
3. 理解 Prompt 的作用与写法
4. 认识大模型 API 的基本调用方式
5. 对比不同风格、不同提示词带来的生成差异

## 4. 使用对象

### 4.1 主要使用者

1. 教师
2. 中等职业学校学生

### 4.2 使用场景

1. 课堂现场演示
2. 学生实验操作
3. Prompt 修改练习
4. AI 文本风格对比教学

## 5. 运行环境

### 5.1 操作系统环境

1. openEuler 22.03
2. 运行在 VMware 虚拟机中

### 5.2 访问方式

1. Linux 虚拟机中运行服务
2. Windows 主机浏览器通过桥接网络访问虚拟机服务

### 5.3 软件环境

1. Python 3.9
2. FastAPI
3. Uvicorn
4. DeepSeek API

## 6. 技术约束

为保证项目适合教学和稳定演示，必须遵守以下约束：

1. 后端框架必须使用 FastAPI
2. 不使用 Flask
3. 不使用数据库
4. 不使用 nginx
5. 不实现登录、支付、权限等复杂业务
6. 不引入 React、Vue 等前端框架
7. 系统以单机运行方式部署
8. 项目代码必须清晰、可运行、便于阅读
9. 尽量减少第三方依赖，避免复杂配置

## 7. 业务范围

### 7.1 本期范围

本项目仅实现一个极简 AI 文本处理网页，功能集中在：

1. 文本改写
2. 风格生成
3. Prompt 展示与编辑
4. AI 结果输出

### 7.2 非本期范围

以下内容不在本项目范围内：

1. 用户登录与注册
2. 数据库存储
3. 历史记录管理
4. 多用户并发权限控制
5. 文件上传下载
6. 复杂管理后台
7. 商业化计费能力

## 8. 功能需求

### 8.1 页面总体要求

系统提供一个单页面 Web 界面，页面布局清晰，适合课堂投屏展示。

页面需包含以下区域：

1. 原始文本输入区域
2. 功能模式选择区域
3. 风格选择区域
4. Prompt 展示与编辑区域
5. 生成按钮区域
6. 输出结果展示区域

### 8.2 功能模式选择

系统应支持以下两种模式：

1. 文本改写
2. 风格生成

#### 8.2.1 文本改写

用户输入一段已有文本，系统根据指定风格与 Prompt 对文本进行改写，输出改写结果。

#### 8.2.2 风格生成

用户输入一个主题、场景或关键词，系统根据指定风格与 Prompt 直接生成一段新文本。

### 8.3 原始文本输入

系统应提供多行文本输入框，用于输入：

1. 待改写原文
2. 待生成主题或关键词

要求：

1. 支持中文输入
2. 可直接粘贴教学示例文本
3. 当用户未输入内容时，可提供默认示例文本

### 8.4 风格选择

系统应提供下拉选择框，供用户选择目标写作风格。

初始版本至少包含以下风格：

1. 正式
2. 小学生
3. 高考作文
4. 鲁迅风

要求：

1. 风格项可在代码中清晰配置
2. 选择不同风格时，默认 Prompt 内容应相应变化

### 8.5 Prompt 展示与编辑

系统必须显式展示当前 Prompt 模板，不允许隐藏 Prompt 拼接逻辑。

要求：

1. Prompt 以文本框形式展示
2. 用户可直接修改 Prompt 内容
3. 用户点击生成时，系统应以当前文本框中的 Prompt 为准调用模型
4. 系统可提供“重置默认 Prompt”按钮，便于教学演示

### 8.6 AI 生成按钮

系统应提供“调用 AI 生成”按钮。

要求：

1. 用户点击后提交当前页面数据
2. 后端接收文本、模式、风格和 Prompt
3. 后端调用 DeepSeek API
4. 结果返回到当前页面显示

### 8.7 输出结果展示

系统应提供多行文本区域显示模型输出内容。

要求：

1. 输出内容清晰可见
2. 支持复制结果
3. 输出区域只读，避免误改

### 8.8 错误提示

系统应在以下情况下给出清晰提示：

1. 未配置 API Key
2. 接口调用失败
3. 网络异常
4. 模型返回异常内容

错误提示要求：

1. 信息简洁明确
2. 便于教师课堂解释问题来源

## 9. 业务流程

### 9.1 正常流程

1. 用户打开网页
2. 页面展示默认示例文本、默认风格和默认 Prompt
3. 用户输入或修改文本内容
4. 用户选择功能模式和风格
5. 系统展示对应的默认 Prompt
6. 用户可继续手动修改 Prompt
7. 用户点击“调用 AI 生成”
8. 后端接收表单数据
9. 后端调用 DeepSeek API
10. 系统将返回结果展示在输出区域

### 9.2 教学演示流程

建议教师按以下顺序演示：

1. 先使用系统默认 Prompt 生成一次
2. 修改风格选项，再生成一次
3. 修改 Prompt 中的一句要求，再生成一次
4. 对比三次结果差异
5. 引导学生总结 Prompt 对结果的影响

## 10. 非功能需求

### 10.1 易用性

1. 页面结构简洁
2. 中文界面清晰
3. 操作步骤少
4. 学生能在一节课内理解

### 10.2 可读性

1. 代码应有清晰注释
2. 文件数量尽量少
3. 函数职责明确
4. 变量命名易懂

### 10.3 稳定性

1. 在 openEuler 22.03 环境中可稳定运行
2. 使用 Uvicorn 本地启动即可完成演示
3. 不依赖数据库或复杂中间件

### 10.4 可维护性

1. 风格配置应集中定义
2. Prompt 模板生成逻辑应集中管理
3. 接口调用逻辑应独立成清晰函数

## 11. 外部接口需求

### 11.1 DeepSeek API

系统通过 DeepSeek 提供的兼容 OpenAI SDK 的接口完成调用。

接口要求：

1. 支持通过 API Key 认证
2. 支持模型名称配置
3. 支持服务地址配置

建议配置项：

1. `DEEPSEEK_API_KEY`
2. `DEEPSEEK_BASE_URL`
3. `DEEPSEEK_MODEL`

## 12. 安全与合规要求

由于本项目是教学 demo，安全要求以基本可控为主。

要求如下：

1. 不在代码中硬编码 API Key
2. API Key 通过环境变量或 `.env` 文件配置
3. 页面不展示敏感密钥
4. 不保存用户输入内容到数据库

## 13. 验收标准

项目验收时，应满足以下标准：

1. 在 openEuler 22.03 中可启动 FastAPI 服务
2. Windows 浏览器可正常访问页面
3. 页面包含文本输入、模式选择、风格选择、Prompt 编辑、生成按钮、结果输出
4. 可成功调用 DeepSeek API 并返回结果
5. 用户修改 Prompt 后，系统按修改后的 Prompt 调用模型
6. API Key 未配置时，页面能正确提示
7. 代码结构简洁，适合教学阅读

## 14. 教学重点标注

本项目推荐重点讲解以下内容：

### 14.1 Prompt 模板生成

讲解如何根据“模式 + 风格 + 原文”拼出默认 Prompt。

### 14.2 表单到后端的数据流

讲解页面表单数据如何通过 POST 请求提交到 FastAPI 路由。

### 14.3 API 调用过程

讲解如何使用 Python 客户端调用 DeepSeek API。

### 14.4 输出结果对比

讲解相同原文在不同 Prompt 下的生成差异。

## 15. 后续可选优化

以下内容不是当前必需，但后续教学中可按需加入：

1. 提供多个课堂示例文本按钮
2. 增加“查看本次请求参数”区域
3. 增加 Prompt 对比示例
4. 增加教师演示用说明区

## 16. 结论

本项目定位为“教学产品”，目标不是构建完整商业系统，而是构建一个简洁、稳定、可讲解、可演示的 AI Web 示例。

项目最终应达到以下效果：

1. 教师容易部署
2. 学生容易理解
3. 课堂容易演示
4. Prompt 作用展示清楚
